# 그날 우리 — 개발 인수인계서 (v2)

> **앱 이름:** 그날 우리 · **스토어 제목:** 그날 우리 : 가족 성장사진 · **프로젝트 폴더명:** `our-day` · **슬로건:** 매달 한 컷, 그날의 우리
>
> 같은 포즈로 매월·매년 가족 사진을 찍어, 시간에 따른 가족의 변화를 기록·비교·공유하는 네이티브 앱.
> 이 문서는 Claude Code에 그대로 전달해 개발을 시작하기 위한 인수인계서입니다.

---

## 0. 한 줄 요약 / 핵심 가치

- **핵심 가치 가설:** "같은 포즈로 꾸준히 찍으면, 시간이 지났을 때 가족의 변화를 보는 감동이 크다."
- **킬러 기능:** 직전 사진을 반투명으로 겹쳐 보여주는 **오버레이 정렬 카메라**. 이게 없으면 컨셉 자체가 성립하지 않음 → 최우선 구현.
- **성패 변수:** 사용 빈도가 낮은(월 1회) 앱이라 **리텐션 설계**가 전부에 가깝다. (6장 참고)
- **v2 변경점:** ① 클라우드 백업(사용자 클라우드 + 소셜 로그인) MVP 포함, ② 위치 기반 회상 알림 추가.

## 1. 기술 스택

- **프레임워크:** Flutter (Dart). 안드로이드를 먼저 출시하고, 동일 코드베이스로 iOS 확장.
- **최소 지원:** Android 8.0(API 26) 이상 / iOS 14 이상 (확장 시).
- **아키텍처:** 단순 계층 분리 — `UI(위젯)` / `상태관리` / `repository(데이터 접근)` / `로컬 DB·파일` / `백업·동기화 서비스` / `위치 서비스`.
  - 상태관리는 `riverpod` 또는 `bloc` 중 하나로 통일.
- **저장 원칙(local-first):** 모든 사진·데이터는 1차로 기기에 저장. 클라우드는 **사용자 본인 클라우드(구글 드라이브 / iCloud)** 로 백업한다. **자체 서버에 사진을 보관하지 않는다.** (8장 참고)

### 주요 패키지 (시작점 — Claude Code가 현재 유지보수되는 최신 버전을 확인 후 채택할 것)
- 카메라: `camera`
- 사진 정렬/처리: `image` (오버레이 합성·리사이즈·간단 정렬 보정)
- 로컬 DB: `drift` 또는 `sqflite`
- 파일 경로: `path_provider`
- 로컬 알림(리마인더): `flutter_local_notifications` + `timezone`
- 타임랩스/슬라이드쇼 영상 생성: `ffmpeg_kit_flutter`(이미지 시퀀스→mp4) — 라이선스/유지보수 상태 확인 필요. 대안 검토 가능.
- 공유: `share_plus`
- 권한 처리: `permission_handler`
- **소셜 로그인:** `google_sign_in`, `sign_in_with_apple`
- **클라우드 백업:** 구글 드라이브 API(`googleapis` + `google_sign_in` 스코프), iOS는 iCloud(파일 기반) 연동
- **위치/지오펜싱:** `geolocator`(위치), `geofencing` 계열 또는 플랫폼 네이티브 지오펜스 브리지(백그라운드 위치)
- **광고:** `google_mobile_ads` (AdMob) + 동의 관리(UMP). 인앱결제/구독은 `in_app_purchase`.

> ⚠️ 패키지는 시간이 지나며 deprecated 될 수 있음. Claude Code는 착수 시 각 패키지의 최신 상태를 점검하고 대체재를 제안할 것. 특히 지오펜싱·백그라운드 위치는 플랫폼별 구현 차이가 크므로 신중히 검증.

## 2. 데이터 모델

로컬 DB 기준. 이미지 원본은 파일시스템에 저장하고, DB에는 경로·메타데이터만 보관. 백업 시 이 구조를 사용자 클라우드에 직렬화/업로드.

### Account (계정 / 로그인)
소셜 로그인 + 본인 클라우드 백업 연결용. 자체 비밀번호는 두지 않음.

| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK (provider uid 기반) |
| provider | enum | `google` / `apple` |
| display_name | string? | 표시명 |
| backup_target | enum | `google_drive` / `icloud` / `none` |
| last_backup_at | datetime? | 마지막 백업 시각 |

### Project (촬영 프로젝트)
하나의 "기록 주제". 예: "우리 가족", "첫째 아이".

| 필드 | 타입 | 설명 |
|---|---|---|
| id | string(uuid) | PK |
| title | string | 프로젝트명 |
| schedule_type | enum | 촬영 주기 유형: `weekly`/`biweekly`/`monthly`/`yearly`/`fixed_dates`/`manual` |
| schedule_config | json | 유형별 상세 (아래 "촬영 주기 설정" 참고) |
| cover_photo_id | string? | 대표 사진 |
| event_peg | enum? | 촬영을 묶을 이벤트: `none`/`birthday`/`holiday`/`season` (6장) |
| created_at | datetime | |

#### 촬영 주기 설정 (사용자가 자유롭게 지정)
주기는 고정값이 아니라 사용자가 프로젝트마다 선택한다. (예: 신생아는 매주, 아이는 매월, 가족 전체는 매년)

| schedule_type | 의미 | schedule_config 예시 |
|---|---|---|
| `weekly` | 매주 | `{ "weekday": 6, "time": "10:00" }` (토요일) |
| `biweekly` | 격주 | `{ "weekday": 6, "anchor_date": "2026-06-01" }` |
| `monthly` | 매월 | `{ "day": 1 }` (매월 1일) |
| `yearly` | 매년 | `{ "month": 6, "day": 23 }` (매년 6/23, 생일·기념일 등) |
| `fixed_dates` | 사용자가 콕 집은 지정일들 | `{ "dates": ["2026-09-15", "2027-01-01"] }` |
| `manual` | 알림 없이 사용자가 원할 때만 | `{}` |

- 알림 기준일/시각은 `schedule_config` 안에서 처리(별도 `reminder_day` 필드 제거).
- "이번 기간" 판정, 진척 게이지(6장 4번), 회상 알림(6장 2번)은 모두 이 `schedule_type`/`schedule_config`를 기준으로 계산.

### Member (구성원, v1.5+)
프로젝트에 속한 가족 구성원. MVP에서는 생략 가능(프로젝트 단위 촬영). v1.5에서 멀티 참여 시 활성화.

| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK |
| project_id | string | FK |
| name | string | |
| role | string? | 예: 엄마/아빠/첫째 |

### Capture (촬영 1건)
| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK |
| project_id | string | FK |
| file_path | string | 원본 이미지 경로 |
| thumb_path | string | 썸네일 경로 |
| captured_at | datetime | 촬영 시각 |
| period_label | string | 표시용 라벨 (예: "2026 · 6월") |
| alignment_meta | json? | 정렬 보정값(이동/스케일/회전) — 사후 재정렬용 |
| note | string? | 한 줄 메모 |
| place_id | string? | FK → Place (위치 기반, 7장) |
| backup_state | enum | `local_only` / `backed_up` / `pending` |

### Place (장소, 위치 기반 기능용)
"사진을 찍은 의미 있는 장소"를 저장. 같은 장소 재방문 시 회상 알림의 기준이 됨.

| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK |
| project_id | string | FK |
| label | string | 장소명 (예: "월미도") — 역지오코딩 또는 사용자 입력 |
| latitude | double | |
| longitude | double | |
| radius_m | int | 지오펜스 반경 (기본 150~300m) |
| capture_count | int | 이 장소에서 찍은 사진 수 |
| geofence_enabled | bool | 이 장소 알림 on/off |

### 인덱스/정렬
- `Capture`는 `project_id` + `captured_at` 시계열 조회가 잦음 → 복합 인덱스.
- `Place`는 `latitude`/`longitude` 근접 조회가 필요 → 위치 인덱싱 또는 단순 거리 계산.

## 3. 화면 명세

MVP는 아래 ①~⑥으로 성립. 각 화면의 목적·핵심 요소·주요 동작만 명시(세부 UI는 구현 단계에서 조정).

### ① 온보딩 / 로그인 / 첫 프로젝트 생성
- **목적:** 진입 장벽 최소화. 로그인 후 30초 내 첫 프로젝트 생성.
- **요소:** 구글/애플 소셜 로그인, 프로젝트명 입력, 촬영 주기 선택(월/년), 이벤트 페그 선택(선택), 권한 요청(카메라·알림·위치).
- **동작:** 로그인 → 백업 대상(드라이브/iCloud) 연결(건너뛰기 가능) → 프로젝트 생성 → 홈으로. 권한은 실제 필요한 시점에 단계적으로 요청(특히 위치는 강요하지 않음).

### ② 홈 / 타임라인
- **목적:** 앱의 중심. "이번 기간 촬영" 유도 + 과거 기록 한눈에.
- **요소:**
  - 상단: 프로젝트명, 설정 진입, 백업 상태 아이콘.
  - **"이번 달(기간) 한 컷" CTA 카드** — 이미 찍었으면 완료 표시, 안 찍었으면 강조.
  - 지난 기록 그리드(썸네일 + 기간 라벨).
  - 진척 게이지(예: 12칸 중 채운 수) — 리텐션 핵심(6장).
  - 하단 탭: 홈 / 앨범 / 공유.
- **동작:** CTA 탭 → 촬영 화면. 썸네일 탭 → 상세/비교.

### ②-1 사진 입력 방식 (구동 방식)
프로젝트에 사진을 넣는 경로는 아래 4가지. 모든 경로의 결과물은 동일하게 `Capture` 1건으로 저장.

1. **앱 내 촬영(기본):** 카메라로 직접 촬영 → 오버레이 정렬 적용 → 저장. (④ 화면)
2. **갤러리에서 불러오기:** 기기에 있는 사진을 선택해 해당 기간에 등록. 불러온 사진에도 정렬 보정(이동·스케일)을 적용해 타임랩스 품질 유지.
3. **과거 일괄 채우기(backfill):** 신규 사용자가 예전 가족 사진을 한 번에 여러 기간에 채워 넣음 → 가입 직후 빈 타임라인 문제 해결, "쌓인 결과"를 즉시 체감(리텐션 매우 중요).
4. **알림에서 바로 촬영:** 리마인더/위치 알림 탭 → 곧장 촬영 화면 진입.

> v1.5에서 **가족 구성원의 공유 기여**(다른 기기에서 같은 프로젝트에 추가)가 5번째 경로로 추가됨.

### ③ 설정 / 백업
- **목적:** 데이터 안전(분실 대비)과 권한 관리.
- **요소:** 로그인 상태, 백업 대상(구글 드라이브/iCloud) 연결·해제, 마지막 백업 시각, 수동 백업/복원 버튼, 자동 백업 토글, 위치 알림 on/off, 앱 잠금(생체인증) 옵션.
- **동작:** 기기 변경/재설치 시 → 로그인 → "복원" → 사용자 클라우드에서 사진·DB 복구.

### ④ 촬영 화면 (★킬러 기능)
- **목적:** 직전 사진과 같은 포즈로 찍게 만든다.
- **요소:**
  - 카메라 실시간 뷰파인더.
  - **직전 사진 반투명 오버레이**(없으면 첫 촬영).
  - 오버레이 **투명도 슬라이더**.
  - 정렬 보조선(가이드 프레임).
  - 셔터 / 갤러리에서 불러오기 / 직전(또는 기준) 사진 토글.
  - (위치 알림으로 진입 시) 그 장소에서 예전에 찍은 사진을 오버레이 기본값으로.
- **동작:** 촬영 → 즉시 미리보기에서 위치·스케일 미세 조정(alignment_meta 저장) → 저장(위치 좌표 기록) → 홈으로.
- **상세 구현은 5장 참고.**

### ⑤ 비교 / 타임랩스
- **목적:** "가치 실현 순간" — 변화를 체감.
- **요소:** 그때 vs 지금 비교 뷰(기간 라벨), 타임랩스 재생, 공유 버튼.
- **동작:** 타임랩스 → 영상 생성 후 재생/내보내기. 4장 미만이어도 짧은 미리보기 제공.

### ⑥ 공유 / 내보내기
- **목적:** 바이럴 루프(6장).
- **요소:** 타임랩스 영상·비교 이미지 내보내기, 단톡방/SNS 공유, (옵션) 앱 워터마크.
- **동작:** 한두 탭으로 공유 완료.

## 4. 킬러 기능 상세 — 오버레이 정렬 카메라

가장 공들여야 하는 부분. 동작 흐름:

1. 새 촬영 진입 시, 해당 프로젝트의 **가장 최근 Capture**(또는 위치 알림으로 진입했다면 **그 장소의 예전 Capture**)를 불러와 뷰파인더 위에 반투명 합성(없으면 가이드 프레임만).
2. 사용자가 투명도 슬라이더로 오버레이 농도 조절(기본 40~50%).
3. 피사체를 오버레이에 맞춘 뒤 촬영.
4. 촬영 직후 미리보기에서 **이동·스케일·회전 미세 조정** → 보정값을 `alignment_meta`에 저장.
5. 저장 시 원본 + 썸네일 생성, 가능하면 위치 좌표 기록.

**품질 포인트**
- 오버레이 기본값은 "직전 사진"이되, 특정 기준 사진을 고정(reference)하는 옵션 제공.
- 타임랩스 생성 시 `alignment_meta`를 반영해 흔들림 감소(얼굴/구도 자동 정렬은 후순위 고도화).

## 5. 위치 기반 회상 알림 (신규)

같은 장소에 다시 갔을 때, 예전에 그곳에서 찍은 사진을 띄우고 다시 찍게 만드는 기능. 시간 기반 알림보다 트리거가 강력함.

### 동작 흐름
1. 사용자가 어떤 장소에서 사진을 찍으면, 그 좌표를 `Place`로 저장(역지오코딩으로 "월미도" 등 라벨 부여, 사용자 수정 가능).
2. 해당 `Place` 주변에 지오펜스(기본 반경 150~300m) 등록.
3. 사용자가 나중에 그 장소 반경에 다시 진입하면 → **"여기, 작년에 가족과 왔던 곳이에요" + 그때 사진 썸네일** 푸시 알림.
4. 알림 탭 → 촬영 화면 진입, 그 장소의 예전 사진을 오버레이 기본값으로 → 같은 구도로 재촬영.

### 설계 주의사항 (중요)
- **백그라운드 위치 권한**은 양 스토어 심사가 까다롭고 사용자 민감도가 높음 → **반드시 선택 기능(opt-in)** 으로, 끄고 켤 수 있게.
- "항상 허용"을 강요하지 말 것. 기능 가치를 설명하고 사용자가 동의할 때만 활성화.
- **배터리 보호:** 모든 위치를 추적하지 말고, 사진을 찍은 의미 있는 `Place` 근처에서만 지오펜스 작동.
- 등록 가능한 지오펜스 수에 플랫폼 한도가 있으므로(특히 iOS), `capture_count` 높은 장소 우선 등 우선순위 관리.
- 알림 빈도 제한(같은 장소 재방문 시 과도한 반복 알림 방지).

## 6. 리텐션 기능 (이 앱의 성패)

월 1회 쓰는 앱이라 "잊혀지지 않게" 만드는 게 핵심. 우선순위순.

1. **이벤트 페그 알림(최우선):** `event_peg`(생일/명절/계절 전환)에 맞춰 "오늘 가족 한 컷 찍기 좋은 날" 알림.
2. **회상형(nostalgia) 알림:** "작년 오늘, 가족은 이런 모습이었어요" + 작년 사진 썸네일 노출 → 감정 트리거.
3. **위치 기반 회상 알림(신규):** 같은 장소 재방문 시 그때 사진 노출 → 즉시 재촬영 유도. (5장)
4. **진척 게이지:** "1년 채우면 타임랩스 완성" 목표를 홈에 진행 막대로. 4컷만 모여도 미리보기 영상.
5. **끊김에 관대한 설계:** 놓친 기간을 나중에 채울 수 있게. 강한 streak 대신 누적 컷 수 강조.
6. **공유 바이럴 루프:** 완성 영상 끝 앱 CTA/워터마크. 공유는 한두 탭.
7. **연말 자동 리캡:** "올해의 우리 가족" 영상 자동 생성 후 푸시.

> MVP에서 최소한 **1·2·3·4·6** 권장.

## 6-1. 광고 영역 / 수익화 설계

수익은 **광고 + 프리미엄 구독(광고 제거 포함)** 의 하이브리드로 간다. 단, 가족·아동 사진이라는 민감한 맥락과 "감성적 기록"이라는 앱 정서를 광고가 해치지 않도록 위치와 형식을 엄격히 제한한다.

### 광고를 넣어도 되는 곳 (허용)
- **홈 / 타임라인 하단의 배너 1개:** 작고 고정된 배너. 콘텐츠를 가리지 않음.
- **앨범 그리드 사이의 네이티브 광고:** 일정 간격마다 1개, 사진처럼 위화감 없는 네이티브 포맷.
- **타임랩스 "생성 완료" 직후의 전면(interstitial) 광고:** 작업이 끝난 자연스러운 휴지점에서만. 빈도 상한(예: 하루 1~2회, n회 생성당 1회)을 둔다.

### 광고를 절대 넣지 않는 곳 (금지)
- **촬영 화면(④):** 셔터를 누르는 결정적 순간엔 광고 없음.
- **회상 알림으로 사진을 보는 순간 / 비교(그때 vs 지금) 뷰:** 감정의 정점이라 광고가 정서를 깬다.
- **아동 사진 바로 위·옆에 밀착된 형태:** 어떤 경우에도 사진과 광고가 한 덩어리로 보이게 두지 않음.
- **전면 광고를 앱 실행 직후나 촬영 흐름 중간에 끼우는 것.**

### 형식·정책 원칙
- 광고 SDK는 **Google AdMob** 기준(`google_mobile_ads`). 시작점이며 최신 상태는 Claude Code가 점검.
- **아동 대상(child-directed) 정책 준수:** 가족·아동 사진 앱은 양 스토어와 광고 네트워크의 아동 관련 정책 적용 가능성이 높음. 광고를 가족 친화 등급으로 필터링하고, 부적절·자극적 광고가 노출되지 않도록 카테고리 차단을 설정. 출시 전 AdMob/스토어의 아동·가족 정책을 반드시 확인.
- 개인화 광고는 동의(consent) 기반으로만. 위치·사진 데이터는 광고 타겟팅에 사용하지 않는다.
- **프리미엄 구독 시 광고 전면 제거** + 고화질 내보내기·무제한 프로젝트·고급 백업 등 제공.

### UI 반영
- 광고 영역은 레이아웃에서 **별도 슬롯 컴포넌트**로 분리해, 프리미엄 사용자/광고 미노출 시 자연스럽게 사라지고 레이아웃이 깨지지 않게 한다.

## 7-1. 사진 인쇄 / 포토북 주문 (v2 유료 기능)

쌓인 사진으로 실물 인화·포토북을 주문하는 기능. 사용자가 사진에 감정적 애착이 크고 이미 콘텐츠가 쌓여 있어 전환율이 높은 수익 모델.

### 방식: 제휴 인쇄 업체 연동 (직접 물류·인쇄 안 함)
- POD(주문형 인쇄) 업체 또는 국내 포토북/인화 업체의 API에 주문을 위임한다. 후보: Printful, Printify 등 글로벌 POD, 또는 국내 인화/포토북 업체. 착수 시 단가·배송·API·국내 배송 지원 여부를 비교해 선정.
- 우리 앱은 **사진 + 주문 정보(상품·수량·배송지)만 전달**하고, 실제 인쇄·결제 정산·배송·CS는 제휴사가 처리.
- 직접 주문 백엔드/물류를 구축하는 방식은 사실상 커머스 사업이 되므로 채택하지 않음.

### 새로 필요한 데이터 (v2에서 추가)
배송·결제가 들어오므로, 기존 "자체 서버에 데이터 안 둠" 원칙과 달리 주문/배송 정보는 처리 주체가 필요하다(제휴사 또는 최소한의 주문 중계). 아래는 데이터 모델 초안.

**Order (주문)**
| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK |
| account_id | string | FK |
| product_type | enum | `print`/`photobook`/`canvas` 등 |
| capture_ids | json | 주문에 포함된 사진들 |
| status | enum | `created`/`paid`/`printing`/`shipped`/`delivered`/`canceled` |
| partner_order_id | string? | 제휴사 측 주문 ID |
| amount | int | 결제 금액 |
| created_at | datetime | |

**ShippingAddress (배송지)**
| 필드 | 타입 | 설명 |
|---|---|---|
| id | string | PK |
| account_id | string | FK |
| recipient | string | 수령인 이름 |
| phone | string | 연락처 |
| address | string | 주소(우편번호 포함) |
| is_default | bool | 기본 배송지 여부 |

### 결제·개인정보 주의사항 (중요)
- **결제:** 디지털 구독(광고 제거 등)은 스토어 인앱결제를 쓰지만, **실물 상품 결제는 외부 PG/제휴사 결제**를 써야 할 수 있음(스토어는 실물 재화에 인앱결제를 요구하지 않음). 출시 전 양 스토어의 실물 상품 결제 정책을 확인.
- **개인정보 제3자 제공:** 배송지·연락처가 제휴 인쇄사로 전달되므로, **명시적 동의 + 개인정보처리방침에 제3자 제공 고지**가 필수. 어떤 정보가 누구에게 왜 가는지 분명히.
- **데이터 최소 보관:** 배송 완료 후 불필요한 배송·연락처 정보의 보관을 최소화.
- 사진은 인쇄 품질을 위해 **원본 해상도 보존**이 필요(아래 "지금 비워둘 자리" 참고).

### 지금(MVP)부터 비워둘 자리
v2에 인쇄를 깔끔히 얹기 위해 MVP 단계에서 미리 대비할 것:
- **원본 해상도 보존:** `Capture`의 원본 이미지를 인쇄 가능한 고해상도로 저장(과도한 다운스케일 금지). 썸네일과 원본을 분리 보관.
- **모듈 격리:** 결제·주문·배송 모듈이 들어올 공간을 구조적으로 분리(`commerce` 계층 자리만 비워둠). MVP에선 구현하지 않음.
- **계정 연결:** 주문은 `Account`에 귀속되므로, 로그인/계정 구조를 주문과 연결 가능하게 설계.

## 8. iOS 확장을 위해 처음부터 지킬 원칙

- **플랫폼 의존 코드 격리:** 카메라·알림·파일저장·공유·**위치·백업(드라이브/iCloud)** 처럼 OS별로 갈리는 부분은 별도 서비스/인터페이스로 추상화해 한곳에 모은다. UI·비즈니스 로직은 플랫폼 무관 유지.

- **권한 처리 분기:** iOS는 권한·푸시·백그라운드 위치 정책이 다름 → 분기 지점 미리 마련.
- **백업 추상화:** 안드로이드는 구글 드라이브, iOS는 iCloud로 갈리므로 백업 서비스를 인터페이스로 두고 플랫폼별 구현 주입.

## 9. 비기능 요구사항 — 프라이버시·보안 (중요)

가족, 특히 **아동의 사진**을 다루므로 신중히 설계.

- **로컬 우선 + 사용자 클라우드 백업:** 사진은 기기에 저장하고, 백업은 **사용자 본인의 구글 드라이브/iCloud**로. **자체 서버에 사진을 보관하지 않는다** → 스토리지 비용·보안 책임·아동 사진 보관 리스크 회피.
- **로그인 최소주의:** 소셜 로그인(구글/애플)만. 자체 비밀번호 시스템 없음.
- **위치 데이터:** opt-in, 언제든 끌 수 있음. 위치는 회상 알림 목적 외 사용·전송 금지. 좌표는 로컬/사용자 클라우드에만.
- **불필요한 수집 금지:** 분석·트래킹 최소화. 사진을 제3자/자체 서버로 전송하지 않음.
- **앱 잠금(옵션):** 생체인증/패스코드로 앨범 보호.
- **공유는 사용자 주도:** 명시적 공유 전엔 어떤 사진·위치도 외부로 나가지 않음.
- **백업 암호화:** 전송·저장 구간 보호.
- **인쇄 주문(v2):** 배송지·연락처가 제휴 인쇄사로 전달되므로 명시적 동의 + 개인정보처리방침 제3자 제공 고지 필수. 인쇄 품질을 위해 사진 원본 해상도는 MVP부터 보존(7-1장).
- **스토어 정책 대비:** 양 스토어의 아동·개인정보·백그라운드 위치·실물 상품 결제 정책을 출시 전 점검(데이터 안전 섹션, 개인정보처리방침, 위치 권한 사유 명시 필수).

## 10. Claude Code 착수 가이드

1. Flutter 프로젝트 초기화, 위 패키지들의 **현재 유지보수 상태/최신 버전 점검** 후 확정.
2. 데이터 모델(2장)부터 DB 스키마·repository 계층 구현.

4. 입력 4경로(②-1) → 로컬 알림(6장 1·2) → 타임랩스(⑤) → 공유(⑥) 순.
5. 소셜 로그인 + 사용자 클라우드 백업/복원 구현(8장 추상화 원칙 준수).

7. 더미 데이터로 전체 플로우 점검 후, 실제 기기에서 카메라·권한·알림·위치·백업 검증.
